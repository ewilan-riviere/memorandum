    ## Making it v-model capable

    Now that our label has been set up, letâ€™s move on to making it so that the component can accept `v-model` bindings.

    Checkboxes have a few quirks that we need to keep in mind before we start setting up our bindings. The first one is that checkbox inputs bind their state to a `checked` property, and not directly to `value`.

    The property `value` of checkboxes is not usually used on the frontend, since its main purpose is to provide a value when submitted to the backend via a submit button. If omitted, this `value` defaults to `on`, which is fine for most apps â€” since most forms will be handled via some asynchronous posting and the passed variables to the backend are controlled by the frontend code.

    The second thing we need to keep in mind, is that inputs with type `checkbox` donâ€™t trigger `input` events, but `change` events whenever they are selected and unselected.

    Now that we are armed with this knowledge, letâ€™s add our `modelValue` prop, which will hold the first part of our two way binding: allowing the parent to inject a state for the checkbox.

    **ğŸ“ƒBaseCheckbox.vue**

    ```javascript

    label: {
    type: String,
    default: ''
    },
    modelValue: {
    type: Boolean,
    default: false
    }
    }
    ```

    Next, letâ€™s go into the `template` and replace the `v-model` binding that was copy-pasted from the demo form earlier with our new `:checked` binding into our `modelValue` prop.

    **ğŸ“ƒBaseCheckbox.vue**

    ```html

        <input type="checkbox" :checked="modelValue" class="field" />

    ```

    Finally, we need to emit `update:modelValue` whenever we want to alert the parent that the checkbox has been toggled. We will listen to the `@change` event on the input element, and emit the new checked state of our element whenever it fires.

    Notice that for checkboxes we are not emitting the targetâ€™s value through `$event.target.value`, but instead the checked status through `$event.target.checked`.
